# Подбор параметров кластеризации и эмбеддингов по размеченному датасету

## Текущее состояние

- **Детекция (прямоугольники)**: YuNet — работает хорошо.
- **Распознавание (персоны)**: ArcFace ONNX (w600k_r50 / arcface_r50_v1) → эмбеддинги; кластеризация DBSCAN (косинусное расстояние) с фиксированными параметрами:
  - `eps=0.4` в [face_scan.py](backend/scripts/tools/face_scan.py), [cluster_face_embeddings](backend/logic/face_recognition.py), в pipeline по умолчанию `--cluster-eps 0.5`;
  - `min_samples=2`.
- **Ground truth для архива**: данные по привязкам после загрузки в архив лежат в привязке через кластеры (ClusterMembers). В текущей схеме таблица `face_cluster_members` удалена миграцией; принадлежность к кластеру хранится в [photo_rectangles.cluster_id](backend/common/db.py), персона кластера — в `face_clusters.person_id`. То есть привязка «лицо → персона» для архива: `pr.cluster_id` → `face_clusters.person_id`. Дополнительно ручные привязки отдельных лиц: [photo_rectangles.manual_person_id](backend/common/db.py).
- **Область для тюнинга**: архив (`f.inventory_scope = 'archive'`), лица с `pr.embedding IS NOT NULL` и `pr.is_face = 1`.

Скриптов, которые оценивают качество кластеризации по ручной разметке, в проекте пока нет.

## Идея

1. **Выборка для оценки**: все лица архива, у которых есть эмбеддинг и определённая персона — через кластер (ClusterMembers: `pr.cluster_id` → `face_clusters.person_id`) или через `pr.manual_person_id`. Ground truth: каждому rectangle_id сопоставляется person_id = COALESCE(pr.manual_person_id, fc.person_id при JOIN по pr.cluster_id).
2. **Метрика**: сравнение разбиения от DBSCAN с разбиением по person_id. Подходят:
  - **Adjusted Rand Index (ARI)** — 1.0 при полном совпадении разбиений, 0 при случайном;
  - дополнительно можно считать **pairwise precision/recall** (доли пар «одна персона — один кластер» и наоборот).
3. **Подбор параметров**: на той же выборке эмбеддингов запускать DBSCAN в памяти с сеткой (eps, min_samples), считать ARI против ground truth, выбрать лучшую пару.
4. **Эмбеддинги**: при желании позже можно добавить поддержку второй модели (например, другой ONNX) и сравнивать ARI при одинаковых параметрах кластеризации.

## План реализации

### 1. Скрипт загрузки ground truth и эмбеддингов

- **Файл**: `backend/scripts/tools/load_face_ground_truth.py` (или в `backend/scripts/debug/`).
- **Действия**:
  - Подключение к БД, выборка по архиву: `photo_rectangles pr JOIN files f ON f.id = pr.file_id WHERE f.inventory_scope = 'archive' AND pr.embedding IS NOT NULL AND pr.is_face = 1`.
  - Персона для каждого rect: приоритет — `pr.manual_person_id`; если NULL — из кластера: `(SELECT fc.person_id FROM face_clusters fc WHERE fc.id = pr.cluster_id LIMIT 1)` (привязки через ClusterMembers).
  - Оставить только строки, где эта персона не NULL.
  - Вернуть/сохранить: список (rectangle_id, person_id), массив эмбеддингов (десериализация из pr.embedding), соответствие индекса массива и rectangle_id.
- **Выход**: либо единый скрипт «оценка + перебор параметров», либо этот шаг — функция внутри следующего скрипта.

### 2. Скрипт оценки кластеризации по ground truth

- **Файл**: `backend/scripts/tools/tune_face_clustering.py` (или `evaluate_face_clustering.py`).
- **Вход**:
  - Опционально: путь к БД, scope (архив / run_id).
  - Сетка параметров: например `--eps-min 0.2 --eps-max 0.6 --eps-step 0.05`, `--min-samples 2 3`.
- **Логика**:
  1. Загрузить ground truth и эмбеддинги (только размеченные лица архива с эмбеддингами).
  2. Нормализовать эмбеддинги (L2), как в [cluster_face_embeddings](backend/logic/face_recognition.py) (строки 288–293).
  3. Для каждой пары (eps, min_samples): вызвать DBSCAN(eps=eps, min_samples=min_samples, metric='cosine'), получить метки кластеров.
  4. Сравнить метки с ground truth (person_id): вычислить ARI (sklearn.metrics.adjusted_rand_score). При необходимости — pairwise F1/precision/recall.
  5. Вывести таблицу (eps, min_samples, ARI, кол-во кластеров, кол-во шума) и рекомендованные параметры (максимум ARI).
- **Зависимости**: numpy, scikit-learn (уже используются в face_recognition). Запуск из окружения с ML (например .venv-face).

### 3. Применение подобранных параметров

- **Кластеризация архива**: сейчас при скане архива в [face_scan.py](backend/scripts/tools/face_scan.py) вызывается `cluster_face_embeddings(..., eps=0.4, min_samples=2)`.
- **Изменения**:
  - Добавить в `clusterize_faces.py` поддержку режима архива (например `--archive`) и передачу eps/min_samples.
  - В `face_scan.py` опционально читать eps/min_samples из аргументов или из конфига (например .env или небольшой конфиг-файл), чтобы после тюнинга можно было запускать с подобранными значениями.
- После тюнинга: подставить выбранные eps (и при необходимости min_samples) в вызовы кластеризации для архива и при желании в pipeline.

### 4. Эмбеддинги (опционально, второй этап)

- Сейчас одна модель: ArcFace (w600k_r50 / arcface_r50_v1), загрузка в [_load_embedding_model](backend/logic/face_recognition.py) и в pipeline.
- Чтобы «подобрать эмбеддинги» под датасет:
  - **Вариант A**: оставить одну модель, тюнить только кластеризацию (eps, min_samples) — уже покрыто шагами 1–3.
  - **Вариант B**: добавить возможность указывать путь к другому ONNX (второй вариант ArcFace/InsightFace), пересчитать эмбеддинги для размеченной выборки (отдельный скрипт или опция в tune), затем запускать тот же перебор eps/min_samples и сравнивать лучший ARI. Это потребует хранения/кэширования эмбеддингов по разным моделям или пересчёта на лету только для выборки оценки.
- Рекомендация: сначала реализовать шаги 1–3 и подобрать eps (и min_samples). Если ARI останется низким — заложить поддержку второй модели и сравнение по ARI.

### 5. Аутлайеры (только в experiments.db)

- **Скрипт** `find_cluster_outliers.py`: читает кластеры из основной БД (только чтение), считает центроид и расстояния; с `--save-to-experiments` пишет список аутлайеров в **experiments.db** (таблица `experiment_outliers`). Основная БД **не меняется**.
- При копировании снапшота (`copy_archive_ground_truth_to_experiments.py`) rectangle_id из `experiment_outliers` исключаются из tune_snapshot — так аутлайеры влияют только на тюнинг в experiments, не на продуктивную разметку.
- В основной БД по-прежнему работает ручная кнопка «Не лицо» (ignore_flag=1) в UI для точечного исключения лиц из центроида.

## Актуализация: параметры, прогоны, хранилище, сравнение

### 1. Какие параметры перебираем и как

- **eps** (порог косинусного расстояния для DBSCAN): основной рычаг. Сетка, например:
  - по умолчанию: от `0.20` до `0.60` шаг `0.05` → 9 значений (0.20, 0.25, …, 0.60);
  - границы и шаг задаются аргументами: `--eps-min`, `--eps-max`, `--eps-step`.
- **min_samples** (минимум точек в кластере): обычно 2 или 2–3. Задаётся списком: `--min-samples 2 3`.
- Итого комбинаций: например 9 × 2 = **18** вариантов (eps × min_samples). Все считаются **в одном запуске** скрипта (см. п. 2).

Дополнительно (второй этап плана): при необходимости можно добавить перебор **модели эмбеддингов** (разные ONNX) — тогда для каждой модели своя выборка эмбеддингов и свой перебор (eps, min_samples).

### 2. Сколько прогонов и нагрузка на сеть (ЯД)

- **Прогонов скрипта: один.** Внутри одного запуска:
  1. **Один раз** читаем из **основной БД**: ground truth (rectangle_id → person_id) и эмбеддинги (`photo_rectangles.embedding` для архива). Файлы с ЯД **не качаем**: эмбеддинги уже лежат в БД (их заполняют скрипты вроде `backfill_archive_manual_embeddings.py` при первичной разметке/бэкфилле).
  2. В памяти для каждой комбинации (eps, min_samples) вызываем DBSCAN по уже загруженным эмбеддингам и считаем ARI.
- **Итог:** забор данных — только чтение из локальной БД; все варианты параметров считаются сразу, без повторных обращений к БД по эмбеддингам и **без загрузки файлов с ЯД**.

### 3. Где храним результаты вычислений

- **Только в БД экспериментов** `backend/scripts/debug/data/experiments.db`. Основная БД (`photosorter.db`) **не пишется** скриптами экспериментов.
- В experiments.db: `tune_snapshot`, `experiment_outliers`, `face_embeddings_alt`, `tune_face_results`. Копирование снапшота из основной БД — чтение; тюнинг и аутлайеры пишут только в experiments.db.

### 4. Как сравниваем опции и вычисляем лучший результат

- **Основная метрика:** **Adjusted Rand Index (ARI)** — сравнение разбиения от DBSCAN с разбиением по ground truth (person_id). ARI = 1.0 при полном совпадении, 0 при случайном.
- **Выбор лучшего:** максимум ARI по сетке (eps, min_samples). При равенстве ARI можно использовать вторичный критерий: меньше шума (точек с меткой -1) или ближе к текущим дефолтам (eps 0.4–0.5, min_samples 2).
- **Дополнительно (опционально):** считать pairwise precision/recall/F1 по парам «одна персона — один кластер» и выводить в отчёт/таблицу для интерпретации.
- Скрипт в конце выводит: таблицу всех комбинаций (eps, min_samples, ARI, n_clusters, n_noise) и явную рекомендацию: «Лучшие параметры: eps=X, min_samples=Y (ARI=Z)».

## Что сделано и почему текущий вариант не оставляем

**Реализовано:**

- Тюнинг по ground truth (tune_face_clustering.py), перебор eps/min_samples, ARI, сохранение результатов.
- Поиск аутлайеров по расстоянию до центроида (find_cluster_outliers.py).
- Все эксперименты вынесены в **отдельную БД** `backend/scripts/debug/data/experiments.db`: снапшот (tune_snapshot), аутлайеры (experiment_outliers), альт-эмбеддинги (face_embeddings_alt), результаты тюнинга (tune_face_results). Основная БД только читается; скрипт reset_ignore_flag_in_main_db.py откатил эксперименты из основной БД.
- Пайплайн run_tune_and_outliers.py (аутлайеры → копирование снапшота → тюнинг), логи в `_logs/`.
- Поддержка второй модели эмбеддингов (backfill в experiments, тюнинг --embedding-source alt --model-key).

**Почему не оставляем текущий вариант:**

- ARI на размеченном архиве ~0.3 — слишком низко для надёжного автоопределения «одна персона».
- Исключение аутлайеров по топ-N/порогу в экспериментах не дало приемлемого качества; при записи в основную БД ситуация ухудшалась.
- Требование пользователя: не считать такой уровень «нормой», искать другой путь.

## Дальнейшие шаги

- **Пункт 1 (вторая модель) — закрыт:** antelopev2 проверена, ARI 0.0131 < 0.0316 (default). Переход на antelopev2 не делаем. Данные antelopev2 в experiments.db можно удалить скриптом `clean_experiments_alt_embeddings.py --model-key antelopev2`.

- **Фильтр по качеству (результат):** в copy_archive добавлены опции `--min-confidence` и `--min-bbox-min`. По тюнингу для архива выбран **min_bbox_min=70**, **eps=0.44** (ARI_clustered ~0.36). В прод внедрены в `clusterize_faces.py --archive` и `face_scan_archive.py`. В `cluster_face_embeddings()` параметр `min_bbox_min` (для архива — только лица с min(bbox_w, bbox_h) >= N). Откат: `--min-bbox-min 40 --eps 0.4`.

- **Пункт 2 — другая кластеризация (сделано):** в тюнер добавлен HDBSCAN (`--algorithm hdbscan`), перебор min_cluster_size (2, 3, 5) и cluster_selection_epsilon (0, 0.1, 0.2). Результат: лучший ARI для HDBSCAN **-0.0373** (min_cluster_size=2, cluster_selection_epsilon=0.2), для DBSCAN — **0.0316**. HDBSCAN на этом датасете не лучше DBSCAN. Оставляем DBSCAN по умолчанию.

- **Аутлайеры в experiments (итог):** скрипт `mark_outliers_in_experiments.py` помечает аутлайеров по расстоянию до центроида персоны (--top-pct или --threshold); при копировании снапшота они исключаются. Прогоны по top-pct: 5% → ARI 0.0433, 10% → 0.0574, 15% → 0.0680, **20% → 0.0756** (лучший), 30% → 0.0457. Порог по расстоянию (--threshold) отсекает почти всех лиц — для экспериментов удобнее --top-pct. Рекомендуемый цикл: copy → при желании mark_outliers (--top-pct 20) → copy → clusterize_in_experiments / tune.

- **Воспроизведение ARI ~0.37:** добавлена метрика **ARI_clustered** (ARI только по точкам, попавшим в кластеры, без шума). Она выводится в `clusterize_in_experiments.py` и в таблице тюнера. При **жёстком фильтре по качеству** (min_bbox_min=70–75) и подобранном eps (0.44–0.46) получаем **ARI_clustered ≈ 0.35–0.37** (полный ARI при этом ~0.22–0.25). Итог: прежний результат ~0.37, скорее всего, был **ARI по точкам в кластерах**, а не полный ARI.

**Рекомендуемые параметры для кластеризации архива:**

| Параметр | Значение | Комментарий |
|----------|----------|-------------|
| **min_bbox_min** | **70** | Учитывать только лица с min(bbox_w, bbox_h) ≥ 70 px (качество кадра). |
| **eps** | **0.44** | Порог косинусного расстояния DBSCAN (альтернатива 0.45–0.46 в том же диапазоне). |
| **min_samples** | **2** | Без изменений. |

**Результаты на снапшоте (min_bbox_min=70, ~1228 лиц, 19 персон):**
- Полный ARI: **0.2503** (при eps=0.46), 0.2426 (при eps=0.44).
- ARI_clustered: **0.3661** (при eps=0.44), 0.3573 (при eps=0.46).

**Точечные эксперименты (проведены):**
- min_bbox_min=70 без аутлайеров: полный ARI 0.2503, ARI_clustered **0.3661** (eps=0.44).
- min_bbox_min=70 + 10% аутлайеров: полный ARI 0.2398, ARI_clustered 0.3286.
- min_bbox_min=70 + 15% аутлайеров: полный ARI 0.2590, ARI_clustered 0.3358.

Вывод: лучший ARI_clustered — без исключения аутлайеров; для полного ARI 15% аутлайеров даёт небольшой прирост (0.259 vs 0.250), но ARI_clustered падает. **Итоговый вариант для прода: min_bbox_min=70, eps=0.44, min_samples=2, без исключения аутлайеров.**

**Встроено в прод:**
- `clusterize_faces.py`: при `--archive` дефолты **min_bbox_min=70**, **eps=0.44** (без флагов — эти значения подставляются автоматически).
- `face_scan_archive.py`: вызов `cluster_face_embeddings(..., eps=0.44, min_samples=2, min_bbox_min=70)` для кластеризации архива.
- Откат к старым значениям: `clusterize_faces.py --archive --min-bbox-min 40 --eps 0.4`.

## Важные детали

- **Ground truth — эталон.** Разметка (привязка лиц к персонам) проверена вручную по каждому фото; это золотой стандарт для оценки. Привязки не игнорируем и не «чистим» — по ним считаем ARI. Лица плохого качества или неудобного ракурса (например только ухо) при необходимости можно исключать **из входа кластеризации** (чтобы не портили центроиды), но для расчёта ARI их метки по-прежнему учитываются — мы сравниваем результат кластеризации с полным ground truth.
- **Размер выборки**: качество оценки зависит от количества размеченных лиц. Если размечено мало персон или мало лиц на персону, ARI может быть нестабильным — в отчёт скрипта стоит выводить число лиц и число персон в ground truth.
- **Шум DBSCAN**: метка -1 (шум) при сравнении с ground truth можно трактовать как отдельный «кластер» для ARI; обычно ARI это корректно учитывает.
- **Приоритет привязок**: для rect в кластере и с manual_person_id использовать manual_person_id; иначе — person_id кластера (ClusterMembers).
